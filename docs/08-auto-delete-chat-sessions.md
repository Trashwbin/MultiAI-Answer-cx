# 自动删除会话设计说明

## 1. 背景

当前插件在调用各家 AI 网页内部接口时，会不断在对应站点产生新的聊天会话。

这会带来两个问题：

- AI 官方网页的聊天列表会堆积大量无用会话
- 后续人工使用这些网站时，历史列表会被插件生成的垃圾会话污染

目标不是提供一个“手动清理”功能，而是：

**插件在拿到有效返回值之后，自动删除本次创建的会话。**

也就是“用完即删”。

## 2. 目标

本设计的目标很明确：

1. 插件发起一次 AI 请求
2. 成功拿到可解析的回答
3. 记录本次会话 ID
4. 在回答返回给插件后，立即调用对应 Provider 的删除接口
5. 不在聊天列表中长期留下插件生成的会话

补充说明：

- 长期目标是覆盖**所有 Provider**
- 但落地顺序必须服从“稳定性优先”
- 也就是说，先把稳定 Provider 的删除做扎实，再决定是否给不稳定 Provider 放开

## 3. 非目标

这次不做下面这些事：

- 不做“清空全部历史会话”入口
- 不做“用户手动选择是否保留本次会话”的复杂设置
- 不优先处理旧 `background.js` 链路
- 不为了删除会话而打断主回答流程

## 4. 当前参考线情况

根据 `Chat2API` 当前实现：

| Provider | 单条删除 | 批量删除 | 备注 |
| --- | --- | --- | --- |
| DeepSeek | 支持 | 支持 | 有 `deleteSession()` 和 `deleteAllChats()` |
| GLM | 支持 | 支持 | 有 `deleteConversation()` 和 `deleteAllChats()` |
| Kimi | 支持 | 未看到 | 有 `deleteConversation()` |

这说明“用完即删”不是空想，至少在 DeepSeek / GLM / Kimi 这几条线上已经有明确参考。

## 5. 我们当前实现现状

### 5.1 DeepSeek

当前我们会显式创建 `chat_session_id`，这意味着：

- 我们能够拿到会话 ID
- 只要补删除接口，就可以在回答结束后回收会话

这是最适合优先接入自动删除的 Provider。

### 5.2 ChatGLM

当前我们虽然是一次请求发送整批题目，但返回里会带会话相关信息。

如果补上删除接口，同样可以做“本次请求结束后自动删除”。

这也是优先级很高的一条。

### 5.3 Kimi

Kimi 这条线理论上也适合接，但前提是：

- 能稳定拿到真实 `chat_id`
- 能在响应完成后安全触发删除

因为 Kimi 本身是 Connect / RPC 协议，删除接入要比普通 REST 更谨慎一点。

### 5.4 ChatGPT / Gemini / Grok

这三条目前不适合作为第一批自动删除目标：

- `ChatGPT`：仍有 API / DOM 双路径，稳定性不足
- `Gemini`：本质上还是纯 DOM 模拟，最脆
- `Grok`：强依赖 anti-bot header 和降级逻辑，长期稳定性不足

因此它们的问题不是“能不能删”，而是：

**主回答链路本身都还不够稳，不该先做会话自动删除。**

## 6. Provider 稳定性分层

在自动删除落地前，先按稳定性把 Provider 分层：

### 6.1 稳定档

建议默认开启：

- DeepSeek
- Qwen CN
- Kimi
- Doubao

这些 Provider 更适合作为第一阶段的主力能力。

### 6.2 观察档

建议保留，但默认关闭：

- GLM

原因：

- 它值得继续做
- 也适合接自动删除
- 但当前实现仍处于“修复后待压实”的状态，不该和稳定档同等待遇

### 6.3 暂停档

建议先禁用：

- ChatGPT
- Gemini
- Grok

原因：

- 当前主链路稳定性不足
- 会话删除并不能解决这些 Provider 的核心问题
- 如果主链路都不稳，再叠加自动删除，只会增加排查复杂度

## 7. 总体策略

这里要明确两件事：

1. **最终目标是所有 Provider 都支持“用完即删”**
2. **真正实施时，必须先把增强做完，再逐步放开**

也就是说，顺序不应该是：

- 所有 Provider 一起补删除

而应该是：

- 先按稳定性分层
- 先禁用不稳定 Provider
- 先完成稳定 Provider 的删除与回收
- 最后再考虑把不稳定 Provider 拉回主流程

## 8. 推荐设计

### 6.1 总原则

自动删除必须是：

- **异步收尾动作**
- **不影响主回答返回**
- **删除失败只记日志，不把整次问答判失败**

也就是说：

1. 先把回答返回给主流程
2. 再尝试删除会话
3. 删除失败不影响用户看到答案

### 6.2 时机

推荐时机：

- **非流式**：`provider.query()` 成功拿到最终文本并完成解析之后
- **流式**：确认流结束、最终答案已组装完成之后

对于我们当前这版插件，建议统一放在：

`query()` 成功返回 `ProviderResponse` 之前或之后立刻触发后台清理任务

但语义上应当保证：

**回答优先，删除其次。**

### 6.3 返回值判定

不要只要请求 200 就删。

推荐删会话的前提：

- 至少拿到了非空 `rawText`
- 或至少解析出一个 `answers[]`

否则如果请求失败、回答为空，却仍然删会话，排查会更困难。

### 6.4 错误策略

删除失败统一按下面处理：

- 控制台打印 warning
- 不抛出到主问答链路
- 不覆盖已有回答结果

错误处理示例原则：

- `query()` 成功，删除失败 -> 本次仍算成功
- `query()` 失败，删除失败 -> 只按原失败返回

## 9. 推荐落地顺序

### 第一批

- DeepSeek
- GLM

原因：

- 参考实现最明确
- 聊天列表污染最明显
- 当前我们已经能拿到必要会话信息

### 第二批

- Kimi

原因：

- 协议更特殊
- 需要先确认 `chat_id` 提取和删除端点稳定性

### 暂缓

- ChatGPT
- Gemini
- Grok
- 其他纯 DOM 或高降级依赖 provider

原因：

- 很多时候没有稳定的结构化会话 ID
- 删除动作可能比主回答本身更脆弱

## 10. 推荐接口形态

建议在 provider 内部补一个可选删除能力：

```ts
interface ProviderSessionCleanup {
  deleteConversation?(sessionId: string): Promise<boolean>;
}
```

然后每个 provider 自己决定：

- 如何提取 sessionId / chatId
- 如何删除
- 删除失败时输出什么日志

主流程不应该知道各家删除端点细节。

## 11. 主流程建议

建议主流程语义如下：

```ts
const result = await provider.query(questions)

if (result.success && result.sessionId) {
  void provider.deleteConversation?.(result.sessionId).catch(logWarning)
}

return result
```

关键点：

- 删除是 `void` 异步尾处理
- 不 `await` 删除结果阻塞主流程
- 不让删除错误污染主回答

## 12. 风险

### 10.1 删除过早

如果在响应流尚未完全结束时就删会话，可能导致：

- 返回内容不完整
- SSE / 流被中断

所以必须等“回答已经拿稳”再删。

### 10.2 删除错会话

如果 sessionId 复用逻辑不清楚，可能误删用户本来就在用的会话。

所以推荐：

- 只删除插件本次显式创建的新会话
- 不删除复用的人工会话

这是一个很重要的保护原则。

### 10.3 DOM 降级模式难一致

像 ChatGPT / Gemini / Grok 这类带 DOM 降级的 provider，删除语义可能并不稳定。

所以第一阶段不建议强推所有 provider 全覆盖。

## 13. 最终建议

这个需求值得做，而且方向应该是：

**不是“加一个清理按钮”，而是“插件生成的会话默认用完即删”。**

推荐实施策略：

1. 最终目标覆盖全部 Provider，但不要一上来全做
2. 先禁用当前不稳定的 Provider：ChatGPT / Gemini / Grok
3. 先对 DeepSeek / GLM 做自动删除
4. 只删除插件新建会话，不删复用会话
5. 删除动作异步执行，不影响主回答
6. Kimi 作为第二批跟进
7. 等主链路足够稳定后，再考虑把暂停档 Provider 拉回并接入删除

## 14. 后续实现备注

真正动代码时，优先检查这几件事：

- 当前 provider 是否能稳定拿到会话 ID
- 当前会话是否为插件新建
- 删除接口是否需要额外签名 / token / chat_id
- 删除动作是否会影响流式回答结束

如果这四点满足，再接入自动删除会更稳。
